# 영웅 시스템 설계

> 상태: 쌓임(누적) 프레임으로 재정립 (2026-04-17 추가 논의) — 메인 죄종 개념 폐기, 특성이 정체성 담당
> 작성일: 2026-03-24 (최종 동기화: 2026-04-27 — Phase C CSV SSOT 분리)

---

## 1. 개요

영웅은 **인간**이다. 용병, 모험가, 떠돌이 — 바알에게 고용된 자들.
각자 깊은 성격적 결함을 안고 있으며, 그 결함이 **죄종 수치**로 표현된다.

멀쩡한 인간은 악마 밑에서 일하지 않는다.
갈 곳 없는 자, 버림받은 자, 망가진 자들이 바알에게 모인다.

### 죄종 수치 = 오염이 쌓이는 차원

**7죄종을 누적 게이지**로 표현 (각 `[balance.csv:sin_min]` ~ `[balance.csv:sin_max]`). 영웅은 **모두 0에서 시작** 하고, 바알 밑에서 행동할수록 특정 죄종이 쌓인다.

- 깨끗 구간 (`0` ~ `[balance.csv:sin_clean_max]`) = 영향 없음 (좋음)
- 쌓일수록 = 파워 보너스 활성 + 문제 행동 위험 동시 증가
- `[balance.csv:sin_rampage_threshold]` 도달 = 폭주 / `[balance.csv:sin_critical_threshold]` 도달 = 임계 이벤트
- 자세한 구간 정의는 → `sin_system.md` §2 "수치 구간"

**"메인 죄종"은 고정 정체성이 아니다.** `topSin()`은 "현재 가장 많이 쌓인 죄종"이라는 상태 표현일 뿐. 영웅의 정체성은 **특성(trait)** 이 담당한다.

| 죄종 수치 | 쌓였을 때 (성격 발현) | 깨끗할 때 |
|---------|--------------------|----------|
| **분노** (공격성) | 충동적, 폭력적, 참을성 없음 | 정상 상태 |
| **시기** (비교심) | 비교 강박, 열등감, 뒷공작 | 정상 상태 |
| **탐욕** (소유욕) | 독점욕, 소유욕, 나누지 못함 | 정상 상태 |
| **나태** (무기력) | 무기력, 회피, 전염되는 체념 | 정상 상태 |
| **폭식** (탐닉) | 과잉, 불안한 비축, 끝없는 허기 | 정상 상태 |
| **색욕** (집착) | 집착, 의존, 관계 중독 | 정상 상태 |
| **교만** (자존) | 자기확신, 독선, 복종 거부 | 정상 상태 |

Phase 1에서는 7체 구현. 죄종 수치 초기값 = 0 (특성의 쌓임 배수에 따라 이후 자연 편향).

## 2. 영웅 구조

```
[영웅] — 반고정 생성 (이름+시작특성 고정, 스탯 랜덤, 죄종 수치 0)
├── 이름 = "수식어" + 이름 (수식어: 쌓인 죄종 상위 2개 조합, hero_epithets.csv)
├── 시작특성 1개 (영웅 이름별 고정, hero_names.csv 매핑) — traits.csv 참조
│   └── 특성이 죄종 쌓임 배수/방향을 결정 (편향/저항/중립)
├── 죄종 7수치 (각 [balance.csv:sin_min]~[balance.csv:sin_max], 단방향 누적, 초기값 0)
├── 기본 스탯 7개 (각 [balance.csv:stat_min]~[balance.csv:stat_max], 랜덤)
├── 파생 세부스탯 — 기본 스탯에서 산출 (행동 + 관계)
├── 후천특성 최대 2개 (Lv5, Lv10 레벨업 시 행동 XP or 폭주 이력 기반)
├── 레벨 (1 ~ 최대치) + XP (카테고리별 누적)
├── acquiredTraits[] — 후천 특성 배열
├── rampageHistory{} — 폭주 유형별 횟수 기록
├── 히든: 성장률 + 상한선 (스탯별)
└── 장비 3슬롯 — Phase 2
```

> **같은 영웅은 1회 플레이에 중복 등장하지 않음.** 21명 풀에서 7명을 뽑으므로 3회차면 전원 조우 → 메타 지식 형성.

> **특성 3슬롯**: 시작특성 1 + 후천특성 2 = 최대 3개. 시작특성이 "정체성", 후천특성이 "플레이어 선택의 성장".

> **UI**: 영웅 고용 시 기본 스탯 + 시작특성 표시. 죄종 수치는 고용 시 전부 0(깨끗). 영입 후 행동에 따라 쌓임.

## 2-1. 스탯 시스템

### 설계 원칙
- **죄종 수치 = 성격**, **기본 스탯 = 능력** — 완전 분리
- 죄종 수치와 기본 스탯 모두 랜덤 생성, 서로 독립
- 한 스탯이 여러 행동에 영향 → 배치 딜레마 발생
- UI에서 행동별 적합도를 별점으로 자동 표시

### 기본 스탯 7개 (각 `[balance.csv:stat_min]` ~ `[balance.csv:stat_max]`)

| 스탯 | 의미 | 영향 |
|------|------|------|
| **힘** | 물리적 파워 | 사냥, 건설, 대장간, 전투 |
| **민첩** | 빠르기, 손재주 | 사냥, 채집, 연금술, 전투 |
| **지능** | 머리, 판단력 | 외교, 연구(단독), 교역, 전투 |
| **건강** | 튼튼함, 면역력 | 체력 최대치/회복 결정 + 발병 저항 |
| **감각** | 눈썰미, 직감 | 채집, 대장간, 연금술 |
| **통솔** | 지휘, 리더십 | 건설, 통솔력(관계), 전투 |
| **매력** | 영향력, 교섭 | 외교, 교역, 고용(비용 할인), 매력도(관계) |

### 체력 시스템

영웅은 **체력(Stamina)** 게이지를 가진다. UI에서 영웅 카드에 표시.

| 항목 | 값 (CSV 키) |
|------|-----|
| **최대치** | `[balance.csv:stamina_max]` |
| **초기값** | 최대치 |
| **턴당 회복** | `[balance.csv:stamina_recovery_base] + 건강 × [balance.csv:stamina_recovery_health_bonus]` |
| **소모** | 행동별 키 (`stamina_cost_gather` / `_lumber` / `_hunt` / `_build` / `_research` / `_feast` / `_expedition`) |

#### 체력 구간별 상태

| 구간 (경계 키) | 상태 | 효과 |
|------|------|------|
| `> stamina_tired_threshold` | 정상 | - |
| `stamina_overwork_threshold` ~ `stamina_tired_threshold` | 피로 | 행동 효율 감소 |
| `0` ~ `stamina_overwork_threshold` | 과로 | 매 턴 발병 확률 체크 (`sickness_chance_base`, 건강 보정) |
| `0` | 탈진 | 행동 불가 |

#### 발병

- 과로 상태에서 매 턴 확률 판정 (`[balance.csv:sickness_chance_base]` - 건강 × `sickness_health_reduction`)
- 발병 지속: `[balance.csv:sickness_min_turns]` ~ `[balance.csv:sickness_max_turns]` 턴 행동 불가
- 발병 시 부수 효과: 현재 `topSin` +1 (없으면 나태 +1)
- 병원 시설 있으면 지속 단축 (미구현)

> 건강 스탯의 의미 = **"체력 최대치/회복 + 발병 저항"**

### 체력 시스템 구현 현황 (2026-04-16)

| 항목 | 상태 | 비고 |
|------|------|------|
| hero.stamina 필드 (1~100, 기본 100) | ✅ | HeroManager._generateHero |
| 턴당 회복 `base + health×bonus` | ✅ | TurnProcessor.processDayPhase |
| 행동 소모 (채집/벌목/사냥/원정) | ✅ | balance.csv `stamina_cost_*` |
| 피로/과로 구간 효율 감소 (채집/벌목 ×0.8/×0.6) | ✅ | DayActions._efficiencyMult |
| 과로 시 매턴 발병 확률 판정 | ✅ | HeroManager.checkSicknessTick |
| 발병 → `status='sick'`, 2~3턴 행동 불가 | ✅ | sickTurns 카운터 |
| UI — MapHeroInspector 체력 바 + 라벨 | ✅ | 정상/피로/과로/탈진 |
| 병원 있을 시 발병 기간 단축 | ⬜ | 미구현 (추후) |

### 전투 내구력

- HP 공식: `[balance.csv:hero_hp_base] + 건강 × [balance.csv:hero_hp_per_vitality]`
- 영웅이 직접 피격. HP 0 → `knocked_out` → 귀환 시 `injured`
- 매 턴 HP 자연 회복: `[balance.csv:hero_hp_regen_per_turn]`. 병원/훈련장/치료약은 시설별 배율 (facilities.csv)

### 죄종 × Stamina 보정

**현재 가장 많이 쌓인 죄종**(`topSin`)을 기준으로 거점 행동 Stamina 소모/회복을 보정. `topSin`은 고정 정체성이 아니라 현재 상태이므로 죄종이 바뀌면 보정도 함께 변함.

| 죄종 | 해석 (서사) | 키 (`balance.csv`) |
|---|---|---|
| 나태 | 쉬엄쉬엄 일하지만 회복은 빠름 | `stamina_mult_sloth_cost` / `stamina_mult_sloth_recover` |
| 분노 | 몰아쳐 일하므로 피로 빨리 누적 | `stamina_mult_wrath_cost` / `stamina_mult_wrath_recover` |
| 시기 | 과업 집착으로 피로 누적 | `stamina_mult_envy_cost` / `stamina_mult_envy_recover` |
| 폭식 | 잘 먹어서 회복↑ | `stamina_mult_gluttony_cost` / `stamina_mult_gluttony_recover` |
| 교만 | 자존심이 휴식을 막음 | `stamina_mult_pride_cost` / `stamina_mult_pride_recover` |
| 탐욕 / 색욕 | 기본 (보정 없음) | `stamina_mult_greed_*` / `stamina_mult_lust_*` |

- 적용 지점: `HeroManager.consumeStamina` / `HeroManager.recoverStaminaTurn` (최고치 죄종 1개 기준)
- **초기엔 모든 죄종이 0이므로 보정 미적용** (수치가 쌓여 `topSin`이 명확해진 후부터 작동)

### Stamina 구간별 효과 (사기 연결 제거 — 2026-04-17)

Stamina 구간은 **행동 효율**에만 영향. 사기 페널티 연결 제거됨.

| 상태 | 조건 (CSV 키 기반) | 효과 |
|------|------|------|
| 정상 | `> stamina_tired_threshold` | 효율 100% |
| 피로 | `stamina_overwork_threshold` ~ `stamina_tired_threshold` | 행동 효율 감소 |
| 과로 | `0` ~ `stamina_overwork_threshold` | 효율 추가 감소 + 매 턴 발병 확률 판정 |
| 탈진 | `0` | 행동 불가 |

적용 지점: `DayActions._efficiencyMult`, `HeroManager.checkSicknessTick`.

### 파생 세부스탯 — 관계 계열

| 세부스탯 | 산출 | 관계에서의 역할 |
|---------|------|---------------|
| **통솔력** | 통솔 | 다른 영웅이 이 영웅을 **따름** (존경) |
| **매력도** | 매력 | 다른 영웅이 이 영웅을 **좋아함** (호감) |
| **민감도** | 색욕 수치 (죄종) | 이 영웅이 다른 영웅에게 **민감하게 반응** |

> 민감도는 기본 스탯 감각이 아닌, 죄종 수치 중 색욕(lust)에서 산출.

## 2-2. 죄종 수치 (쌓임 시스템)

### 설계 원칙
- **7죄종 누적 게이지** (각 0~20, 단방향 상승)
- 초기값 **전부 0** (깨끗한 상태) — 행동에 따라 쌓임
- **쌓일수록 파워 보너스 + 문제 행동 위험** 동시 증가 (양면성)
- **"메인 죄종"은 고정 정체성 없음** — `topSin()`은 현재 최고 수치라는 상태일 뿐
- 정체성은 **시작특성**이 담당 (`[다혈질]`이면 분노가 쉽게 쌓임, 결과적으로 분노가 topSin이 됨)
- 관계 매트릭스(동류/강화/중립/대립)는 현재 `topSin()` 기준으로 활성화 (`sin_system.md` §3 참조)

### 7개 죄종 수치 (쌓였을 때 발현 효과)

| 죄종 수치 | 쌓였을 때 | 상세는 TBD |
|---------|----------|----------|
| **분노** (공격성) | 전투 공격력↑, 동료 충돌 잦음, 18+ 도발 행동 | 파워 보너스 수치 |
| **탐욕** (소유욕) | 보상 획득량↑, 18+ 자원 횡령 | 파워 보너스 수치 |
| **교만** (자존) | 리더 시 능력↑, 18+ 명령 거부/이의 | 파워 보너스 수치 |
| **시기** (비교심) | 탐색/연구 보너스, 18+ 강한 영웅 약화 | 파워 보너스 수치 |
| **폭식** (탐닉) | 장기 작업 보너스, 18+ 식량 추가 소비 | 파워 보너스 수치 |
| **색욕** (집착) | 관계 형성 빠름, 18+ 배치 거부 | 파워 보너스 수치 |
| **나태** (무기력) | 혼자 행동 효율↑, 18+ 효율 감소+전파 | 파워 보너스 수치 |

> 구간별 경계값(파워 보너스 활성 시점 등)은 다음 세션 확정.

### 폭주/이탈 행동 결정 방식

1. **행동 강박 특성 우선** — 후천 폭주 특성(파괴자/약탈자 등)이 있으면 해당 강박 행동 우선 발동
2. **쌓인 죄종 기반** — 특성 예외가 없으면, **18+ 도달한 죄종**의 문제 행동 발동
3. 여러 죄종이 동시에 18+인 경우의 우선순위는 세부 규칙 TBD

> 수식어(이름 생성)는 **현재 쌓인 상위 2개 죄종 조합**. 게임 진행에 따라 수식어가 바뀔 수 있음.
> 예: 분노+탐욕이 가장 높게 쌓이면 "약탈자" 수식어

## 2-3. 특성 시스템 (3슬롯 구조)

### 개요
영웅은 **최대 3개 특성**을 가진다 — 시작특성 1 + 후천특성 2.

```
시작특성 1개 (영웅 이름별 고정, hero_names.csv 매핑) — 정체성 + 죄종 쌓임 편향
+ 후천특성 1개 (Lv5 도달 시, 행동 XP 기반 or 폭주 이력 기반)
+ 후천특성 1개 (Lv10 도달 시, 행동 XP 기반 or 폭주 이력 기반)
= 최대 3개
```

> **명명**: 기존 "고유 특성" 용어를 "시작특성"으로 개념 통일 (내용은 동일, 이름별 고정 매핑 유지).
> **정체성 담당**: "메인 죄종 = 정체성" 구조를 폐기하면서 **시작특성이 영웅의 정체성**을 담당. 죄종 수치는 모두 0에서 시작하므로, 시작특성이 "어떤 죄종이 쉽게 쌓이는지"를 결정 → 자연스럽게 해당 죄종이 topSin이 됨.

## 2-4. 성장 시스템 (레벨 + XP)

### 개요
행동을 할 때마다 **경험치(XP)**를 얻고, 총 XP가 쌓이면 **레벨업**한다.
레벨업 시 **가장 많이 쌓인 XP 카테고리**의 관련 스탯이 상승한다.
최대 레벨은 **10**.

### XP 카테고리

| XP 카테고리 | 획득 행동 | 레벨업 시 상승 스탯 |
|-----------|----------|-----------------|
| 전투 XP | 원정, 방어전 전투 | 힘, 민첩 |
| 방어 XP | 방어전 배치 | 건강, 통솔 |
| 노동 XP | 채집, 벌목 | 민첩, 감각 |
| 건설 XP | 건설 | 힘, 통솔 |
| 연구 XP | 연구 | 지능 |
| 사냥 XP | 사냥 | 힘, 민첩 |

### 레벨업 흐름

```
행동 → 해당 카테고리 XP 획득 → 총 XP 누적
  → 레벨업 임계값 도달 → 레벨 상승
    → XP 1위 카테고리의 관련 스탯 +1~2
    → Lv5: 후천 특성 1개 획득
    → Lv10: 후천 특성 1개 획득 (총 2개)
```

### 레벨업 보상

| 레벨 | 보상 |
|------|------|
| Lv2~4, 6~9 | XP 1위 카테고리 관련 스탯 +1 |
| **Lv5** | 스탯 +1 + **후천 특성 1개** (폭주 이력 우선 → 행동 XP 기반) |
| **Lv10** | 스탯 +1 + **후천 특성 1개** (동일 로직, 2위 카테고리) |

### 후천 특성 획득 판정 (Lv5, Lv10)

```
1. 폭주 이력 2회+인 유형이 있는가?
   → YES: 해당 폭주 특성 부여 (행동 특성 대신)
   → NO: 행동 XP 1위 카테고리의 후천 특성 부여
2. Lv10에서는 남은 슬롯에 같은 로직 반복 (2위 카테고리 or 2번째 폭주)
```

> 부수 효과: 폭주가 반복되면 후천 슬롯 일부가 폭주 특성으로 채워져, 행동 XP 보상이 그만큼 줄어든다.

### 히든 파라미터 (플레이어에게 비공개)

| 파라미터 | 의미 | 비고 |
|---------|------|------|
| **성장률** (스탯별) | 해당 스탯 상승 속도 | 같은 행동을 해도 영웅마다 성장 속도 다름 |
| **상한선** (스탯별) | 해당 스탯 최대 도달치 | 재능의 한계 — 상한이 20 이상이면 초월 성장 가능 |

### 딜레마 예시

```
원정만 보내던 영웅이 Lv5 도달
  → 전투 XP 1위 → "역전의 용사" 획득!
  → 하지만 폭주 2회 경험했다면 → "파괴자" 획득... 전투 보상 날림

건설+연구를 골고루 시킨 영웅이 Lv5
  → 건설 XP가 근소하게 1위 → "건축가" 획득
  → Lv10에서는 연구 XP가 2위 → "연구자" 획득 → 만능형 성장

채집만 시킨 영웅이 Lv5, Lv10 모두 노동 XP 1위
  → Lv5 "노동의 기쁨" → Lv10은 2위 카테고리 특성
```

## 3. Phase 1 영웅 (7체)

영웅은 **반고정 생성**. 이름+시작특성은 고정, 스탯은 랜덤. 죄종 수치는 전부 0에서 시작. 직업 없음 — 스탯과 장비가 역할을 정의.
- 이름: 21명 고정 풀에서 선택 (중복 불가) + 쌓인 죄종 상위 2개 조합 수식어 (진행 중 갱신)
- 시작특성: 영웅 이름별 1:1 고정 (hero_names.csv 매핑, 장점 전용)
- **죄종 수치: 7개 전부 0에서 시작** (행동에 따라 쌓임)
- 기본 스탯: 7개 각 1~20 랜덤
- 레벨: 1 (최대 10), XP: 카테고리별 0
- 후천특성: Lv5, Lv10에서 각 1개 획득 (행동 XP or 폭주 이력 기반)

## 2-5. 이름 수식어 시스템

### 개요
영웅 이름은 **"수식어" 이름** 형태 (예: "뼈칼" 밀드, "고독한" 리아).
수식어는 영웅의 **죄종 수치 상위 2개 조합**(7C2 = 21가지)에 의해 결정된다.

### 수식어 테이블 (21 조합 × 3 후보 = 63개)

> **데이터 SSOT**: `src/data/hero_epithets.csv` (stat1, stat2, epithet1, epithet2, epithet3 — 21행).
> 본문에는 카탈로그를 노출하지 않음 — 이름/조합 의미는 아래 게임플레이 의미만 유지.

### 규칙 (쌓임 프레임 반영)
- **초기엔 수식어 없음 또는 "이름 없는"** — 모든 죄종이 0이라 상위 2개 추출 불가
- 죄종이 쌓여 의미 있는 수치가 형성되면 **상위 2개 조합 기반 수식어 부여**
- 수식어는 **플레이 중 변화 가능** (쌓이는 죄종이 바뀌면 갱신)
- 이름 형식: `"수식어" 이름` (예: "피바람" 타론)
- 데이터: `src/data/hero_epithets.csv`

### 게임플레이 의미
- 수식어 = **현재 쌓인 죄종 상위 2개의 교차점** (상태 표현)
- "약탈자" = 분노 + 탐욕 많이 쌓임 → 전투 강하고 보상 많지만, 폭주 위험
- "고독한" = 색욕 + 나태 많이 쌓임 → 혼자 강하지만 관계에 흔들림
- **수식어 변화 = 영웅의 "지금 상태"** → 서사적 변신감

## 6. 영웅 관계 시스템 (bonds) — 요약

> **상세 SSOT**: [sin_system.md](sin_system.md) §2-A (계산 순서·관계→죄종 증폭) + §3 (관계 매트릭스).
> 본 절은 영웅 데이터 구조 관점의 요약만 둠.

### 데이터 구조
```js
hero.bonds = { [otherHeroId]: bonds_min..bonds_max }  // asymmetric, 초기값 [balance.csv:bonds_initial]
hero.sinPriority = [...7죄종 셔플]                     // 생성 시 영구 고정, 동률 해체용
```

- `bonds`: A→B와 B→A가 **다를 수 있음** (짝사랑/일방적 적대)
- `sinPriority`: **숨은 기질** — `topSin()` 동률 시 1순위 채택
- 바알↔영웅 친밀도는 별도 축 없음 — 후천 특성(`acquiredTraits`)으로 이산 기록
- 핵심 원칙: 영웅은 봉신 아닌 용병 → 7감정(죄종 수치)이 본질이고, 관계 변화는 죄종 쌓임 증폭/억제로 반영

## 7. 낮 상태 · 밤 방어 구조 — 2026-04-20 확정

### 낮 상태 3분류

```
[낮 상태]
├── 작업 중       → 생산적 행동, Stamina 소모
│    ├── 거점 내 (건설/연구/생산/채집) → 밤 방어 O
│    └── 거점 외 (원정/벌목/사냥)     → 밤 방어 X
├── idle          → Stamina 회복 (숙소 품질 배율), 밤 방어 O
└── 특수 행동     → 전용 효과, 밤 방어 X
     └── 병원 입원 (부상 전용, 치유 속도 가속)
```

- **"휴식"과 "idle" 병합** — 별도 상태 아님
- **격리실/수련/참회** 등 특수 행동 후보는 Phase 2 이후 확장

### 밤 방어 자동 판정

```js
canDefend(hero) = 
  hero.location === 'base'           // 거점 내
  && hero.stamina >= threshold        // 스태미너 임계 이상
  && !isInjured(hero)                 // 부상 아님
  && !isInSpecialAction(hero)         // 특수 행동 중 아님
```

- **명시적 "방어 배치" slot 제거** — Stamina가 자동 게이트
- **부상 영웅**: 뭘 하든 방어 불가, 병원으로 회복 가속 가능
- 임계 키: 방어 가능 Stamina = `[balance.csv:stamina_defense_threshold]` 이상, 원정 1회 소모 = `[balance.csv:stamina_cost_expedition]`
